## Day 2：HAL层 + RPC服务层（2.5小时）

### 重点文件
- lesson5/rpc_server/hal.h、hal.c
- lesson5/rpc_server/rpc_server.c
- lesson5/rpc_server/dht11.c、led.c

### 课程关联 + 深度回答

**Q: HAL层的设计跟你学的哪门课相关？**（→ 软件工程 + C语言）

软件工程课上讲"高内聚低耦合"，HAL层就是典型的例子。所有硬件操作集中在hal.c，上层rpc_server.c不知道GPIO是sysfs还是/dev——它只调用hal_xxx()函数。我用了C语言的**函数指针**（hal.h中定义的回调类型）和**结构体封装**（hal_config_t配置结构体），这些都是C语言课上的知识点。

具体实现上，每个传感器用内部状态机管理：
```c
typedef struct {
    int value;                // 当前值
    int failure_count;        // 连续失败次数
    hal_sensor_status_t status; // ONLINE/OFFLINE
    time_t last_success;      // 上次成功时间
    time_t retry_interval;    // 恢复重试间隔（60秒）
} sensor_state_t;
```
读取失败时failure_count++，达到5次标记OFFLINE，60秒后自动尝试恢复。这个"失败计数→状态切换→自动恢复"的模式，其实是操作系统课上学到的**有限状态机**思想。

**Q: 滑动平均滤波和防抖对应什么课程知识？**（→ 数据结构 + 单片机）

滑动平均滤波用了**循环队列**的数据结构：
```c
static int temp_buf[TEMP_FILTER_WINDOW] = {0};
static int temp_idx = 0;
// 每次新值覆盖最旧数据，然后取平均
int sum = 0;
for (int i = 0; i < window; i++) sum += buf[i];
return sum / window;
```
这是数据结构课上学的最基础的数组实现循环队列。温度/湿度窗口3、光照窗口5——这个参数选择是我根据传感器噪声特性调的，噪声大的窗口大一些。

烟雾防抖用了**连续采样确认**的方法：连续3次读到同一状态才确认变化。这其实是单片机课上学到的**按键消抖**思想——机械按键按下时有抖动，GPIO输入也有尖峰干扰，3次确认可以在15ms内完成判断，既保证响应速度又滤除干扰。

**Q: DHT11的驱动方式对应什么知识？**（→ 嵌入式系统 + 操作系统）

DHT11的数据手册定义了单总线协议：主机拉低18ms发起起始信号，DHT11响应后发送40位数据（湿度整+湿小+温度整+温小+CRC），每位数据用不同的脉宽表示0或1。

我们用一个后台线程轮询/dev/mydht11设备，结果存在全局变量中：
```c
// dht11_thread: 后台每1秒读一次
while (1) {
    if (2 == read(fd, buf, 2)) { g_humi = buf[0]; g_temp = buf[1]; }
    sleep(1);
}
// dht11_read: 直接返回共享内存，O(1)无阻塞
*humi = g_humi; *temp = g_temp; return 0;
```
这个"生产者-消费者"模式对应**操作系统课上的线程同步**——内核驱动是生产者（写入数据），rpc_server是消费者（读取数据）。之所以用后台线程而不是每次发起通信，是因为DHT11单总线协议一次通信要约20-25ms（拉低18ms起始+数据传输），如果每次RPC调用都发起通信会阻塞调用方。

**Q: 9个RPC方法对应什么课程知识？**（→ 计算机网络）

**计算机网络课**上讲应用层协议设计。JSON-RPC是一种轻量级的远程过程调用协议，请求格式是JSON：
```json
{"method": "led_control", "params": [1], "id": "1"}
```
选择JSON-RPC而不是gRPC的原因：i.MX6ULL是ARM Cortex-A7 @528MHz，gRPC需要C++运行时+protobuf，静态编译后体积大10倍。JSON-RPC用jsonrpc-c库（纯C），编译后约200KB，内存占用小。这是**嵌入式系统课**上讲过的资源权衡——在有限资源下选择合适的协议。

## Day 3：MQTT网关 上 — 线程模型（2.5小时）

### 重点文件
- lesson6/mqtt_bridge.cpp（分段读关键区间）
- lesson6/rpc_client.h/cpp

### 课程关联 + 深度回答

**Q: mqtt_bridge的4个线程对应什么课程知识？**（→ 操作系统）

**操作系统课**上讲的**多线程编程**和**线程同步**。4个线程各有分工：
- **主线程**：初始化完成后进入while循环每秒喂狗
- **MQTT网络线程**：mqttclient库内部创建，维护TCP连接和心跳
- **命令工作线程**：从消息队列接收命令，调用RPC执行硬件操作
- **遥测线程**（5秒周期）：读取传感器→事件驱动上报→边缘自动控制

这里用到了操作系统的两个核心机制：**互斥锁**（pthread_mutex_t保护fan_state/led_state）和**条件变量/消息队列**（生产者-消费者模型解耦MQTT回调和RPC执行）。

**Q: 为什么MQTT回调不能直接调用RPC？**（→ 计算机网络 + 操作系统）

MQTT网络线程负责维持TCP连接和发送心跳包（pingreq/pingresp）。如果回调中执行耗时操作（RPC调用涉及TCP通信到rpc_server），网络线程被阻塞，心跳不能及时发送，broker会在超时后判定设备离线。这是我们用消息队列解耦的原因：
```
MQTT回调（生产者）→ msgq_try_send() → cmd_queue → msgq_receive() → 命令工作线程（消费者）
```
这是**操作系统课**上讲过的**生产者-消费者模型**，用队列缓冲解决生产速度和消费速度不匹配的问题。

**Q: 消息优先级怎么实现的？**（→ 数据结构 + 操作系统）

插入时按优先级排序，同优先级FIFO。smoke和restart命令用URGENT优先级，确保安全关键消息优先处理。这是**数据结构课**上讲过的**优先队列**概念——虽然这里用数组遍历实现（数据量小性能足够），不是用堆。

**Q: MQTT协议用到了网络课的哪些知识？**（→ 计算机网络）

**计算机网络课**上讲TCP/IP协议栈，MQTT运行在TCP之上，项目里用了QoS0（心跳，最多发一次）和QoS1（遥测/控制/告警，至少一次）。QoS1保证消息至少到达一次，但连接期间断开会丢——这个问题我们通过应用层环形缓冲区补偿了（断网缓存，恢复后重传），用**应用层可靠性保证**弥补**传输层可靠性不足**。这对应**计算机网络课的端到端设计原则**。

**Q: 断网缓存的数据结构是什么？**（→ 数据结构）

**数据结构课**上讲过的**环形缓冲区**（circular buffer）。固定容量100条，满时覆盖最旧数据，初始化时从文件恢复。用环形缓冲区而不是链表的原因是：固定大小，没有动态内存分配（嵌入式环境碎片问题），O(1)入队出队。这是数据结构课教的"不同场景选不同数据结构"的思路。

## Day 4：MQTT网关 下 — 自动控制 + 图片（2.5小时）

### 重点文件
- lesson6/mqtt_bridge.cpp：auto_control_xxx()和publish_image_xxx()

### 课程关联 + 深度回答

**Q: 三级自动控制对应什么课程知识？**（→ 物联网技术 + 嵌入式系统）

**物联网技术课**上讲**边缘计算**——在靠近数据源的地方做决策，而不是把数据全传到云端再处理。三级控制（烟雾>温度>光照+PIR）全在本地执行，不依赖云端。即使网络断开，自动控制也能正常运行。上层更紧急的控制抑制下层控制，防止冲突——这个"优先级调度"的思想来自**操作系统课**的任务调度。

**Q: 继电器跳匝问题的根因分析对应什么能力？**（→ 程序调试综合能力）

这个问题分三层分析：

第一层（现象）：温度>32度时风扇能吹，但继电器每5-30秒咔嗒响。
第二层（代码分析）：auto_control_smoke中判断`now >= *smoke_fan_until`，smoke_fan_until初始值0，now（当前时间戳，约17亿）>=0永远为真。所以无烟雾时每5秒关一次风扇，紧接着温度联动又打开。
第三层（系统思考）：连续报警时30秒定时器到期后，烟雾联动关风扇，温度联动发现温度仍高又打开。两个控制逻辑各干各的，没有协调。

修复三点：smoke_fan_until>0守卫、到期查温度、处理完重置。同时给fan_state/led_state加了互斥锁。
**这是软件工程课上的调试方法论**：从现象→定位根因→设计修复→验证效果。

**Q: 温度控制为什么用滞回？**（→ 自动控制原理）

两个阈值：32度开风扇，30度关风扇，中间2度滞回区。如果没有滞回（单一阈值32度），温度在31.9~32.1度波动时继电器会频繁开关。**自动控制原理课**上讲**滞回比较器**，就是用来解决这种阈值附近振荡问题的。

**Q: 图片双通道传输对应什么课程知识？**（→ 计算机网络）

**计算机网络课**上讲**可靠传输**——没有一种传输方式能在所有条件下可靠，所以用主备方案。HTTP POST传输效率高（无Base64编码开销），但5秒超时可能失败。MQTT Base64保证能送达（QoS1），但有64KB限制和33%编码开销。主备双通道自动切换，这是**网络课讲过的冗余设计思想**。

**Q: 图片上传改成异步线程对应什么知识？**（→ 操作系统）

原来的HTTP上传阻塞遥测线程（5秒超时），我改成创建独立线程：
```c
pthread_t upload_tid;
pthread_create(&upload_tid, NULL, image_upload_thread, task);
pthread_detach(upload_tid);
```
**操作系统课**上讲**异步I/O**——不能让一个慢操作阻塞整个周期。这里用独立线程实现异步，不上传的时候不影响传感器采集和自动控制。pthread_detach让线程结束时自动释放资源，不需要join等待。

## Day 5：Web API + 云端（2小时）

### 重点文件
- lesson5/rpc_server/web_api.c：认证流程
- cloud/mqtt_to_influxdb.py：MQTT回调+钉钉

### 课程关联 + 深度回答

**Q: Web认证用了什么加密知识？**（→ 网络安全/C语言）

密码用SHA-256加盐哈希存储（盐是hostname+machine，确保两台设备同密码也不同哈希）。Token用/dev/urandom生成。这对应**网络安全课**上讲过的**密码存储最佳实践**：不存明文、加盐哈希、用安全随机数。

**Q: Token生成改过什么？**（→ 网络安全）

原来用srand(time(NULL) + getpid())，time(NULL)精度到秒，getpid()也只有几位——攻击者知道设备大概启动时间就能预测Token。我改成了从/dev/urandom读随机数。这是网络安全课强调的**密码学安全随机数发生器**和普通随机数的区别。

**Q: 云端用到了什么技术？**（→ 物联网技术）

云端部署在阿里云，Python脚本订阅MQTT主题，写入InfluxDB时序数据库。InfluxDB是专门存时序数据的（带时间戳的传感器数据），用SQL-like查询语言。数据可视化用Grafana——自动配置仪表板显示温湿度趋势、传感器状态、CPU负载。这套技术栈是物联网技术课讲到过的**云-管-端**架构中的"云"部分。
