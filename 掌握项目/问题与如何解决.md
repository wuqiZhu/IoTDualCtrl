## 一、硬件与驱动问题

### 1.1 摄像头首帧黑屏

**现象：** 每次抓拍的第一张照片是纯黑的，或颜色严重偏色。

**根因：** USB 摄像头刚初始化时，自动曝光（AE）和自动白平衡（AWB）算法需要若干帧才能收敛。第一帧数据通常是摄像头 sensor 的原始未处理状态，芯片 ISP 还没有完成图像处理 pipeline。

**解决：** 丢弃前 5 帧，取稳定后的帧保存。

```c
for (int i = 0; i < 5; i++) {      // 丢弃前5帧
    camera_get_frame(&discard);
    camera_release_frame(&discard);  // 直接归还缓冲区
}
camera_get_frame(&frame);           // 取第6帧，稳定的
fwrite(frame.data, ...);
```

**为什么不直接 sleep？** sleep 时间不好确定（不同摄像头启动时间不同），用帧计数法自适应。

---

### 1.2 摄像头设备独占冲突

**现象：** Web 页面抓拍有时成功有时失败，烟雾报警拍照也经常拍不到。

**根因：** `/dev/video1`（V4L2 设备）只能被一个进程打开。mqtt_bridge 初始化时占住设备，Web API 调用时无法打开。

**解决：** 改由 rpc_server 统一管理摄像头，mqtt_bridge 通过 RPC 调用拍照。

```
修复前:
  mqtt_bridge:  init 时占住 /dev/video1 不释放 → rpc_server 打不开

修复后:
  rpc_server:  临时 init → capture → cleanup，用完即释放
  mqtt_bridge: 调用 rpc_camera_capture_jpeg() 通过 RPC 拍照
```

---

### 1.3 DHT11 驱动读取偶发失败

**现象：** DHT11 温湿度读取偶尔返回 -1（无效值），尤其在连续读取时。

**根因：** DHT11 单总线协议对时序要求严格。连续读取间隔太短（<1秒）时传感器未准备好，或中断被更高优先级中断打断导致脉宽测量不准。

**解决：** 加入重试机制（最大 10 次，间隔 1ms），并在 HAL 层做滑动平均滤波。

```c
while (dht11_read(&humi, &temp) != 0) {
    if (++retry_count >= 10) return HAL_ERROR_TIMEOUT;
    usleep(1000);  // 等1ms重试
}
// 滑动平均窗口3，滤除偶发尖峰
temp = sliding_average(temp_buf, &temp_idx, &temp_count, 3, temp);
```

### 1.4 ARM 平台 char 符号性导致的值判断错误

**现象：** DHT11 读取的温度值在某些情况下判断错误。

**根因：** ARM 平台 `char` 默认是 `unsigned char`（0~255），而 x86 默认是 `signed char`（-128~127）。代码中用 `(char)-1` 判断错误，在 ARM 上 `(char)-1` 实际是 `255`，与 `-1` 比较永远不相等。

**解决：** 统一使用 `int` 类型存储传感器值，只在驱动层最底层使用 `char`，读出来立即转 `int`。

```c
int humi, temp;
char raw_humi, raw_temp;
dht11_read(&raw_humi, &raw_temp);
humi = (int)(unsigned char)raw_humi;  // 明确转 unsigned
temp = (int)(unsigned char)raw_temp;  // 避免 ARM 和 x86 差异
```

## 二、软件架构与竞态问题

### 2.1 继电器跳匝（最高频问题）

**现象：** 温度超过 32°C 时风扇能吹，但继电器每 5~30 秒跳一次（咔嗒声），一天能跳 15 次以上。

**根因：** 两个独立控制逻辑互相冲突。

**根因①：`smoke_fan_until` 初始值为 0**

```c
if (fan_state && now >= *smoke_fan_until)
// smoke_fan_until = 0, now ≈ 17亿（当前时间戳）
// → 17亿 >= 0 永远为真！从未报警也每5秒关一次风扇
```

**根因②：烟雾定时器到期后与温度控制打架**

```
烟雾报警 → smoke_fan_until = now + 30
30秒到期 → auto_control_smoke 关风扇
        → auto_control_temp 发现 temp>32°C → 又开风扇
        → 继电器每5秒重复一次这个循环
```

**解决：**

```c
// auto_control_smoke else 分支——三点修复
if (fan_state && *smoke_fan_until > 0             // ① 守卫：没报过警不介入
    && now >= *smoke_fan_until) {
    if (g_sensor_cache.temp < temp_high_threshold) {
        rpc_relay_control(0);   // 关风扇
        fan_state = 0;
    }
    // ② 温度仍高：不关，交给温度联动管理
    *smoke_fan_until = 0;        // ③ 重置，防止下次循环重复触发
}
```

**结果：** 跳匝频率从 15+ 次/天降到 ≤1 次/天，改善 92%。

### 2.2 温度控制阈值附近振荡

**现象：** 温度在 32°C 附近波动时，继电器频繁开关。

**根因：** 单一阈值（如 >32°C 开、≤32°C 关）会导致阈值附近反复切换。

**解决：** 引入滞回控制（hysteresis），两个阈值。

```c
// 温度 >32°C → 开风扇
if (temp > 32 && !fan_state) rpc_relay_control(1);
// 温度 <30°C → 关风扇（中间2°C滞回区）
if (temp < 30 && fan_state) rpc_relay_control(0);
```

**滞回区 30~32°C**：在这个区间内风扇保持当前状态不变，自然消除了阈值附近振荡。

---

### 2.3 Web 前端图片缓存不更新

**现象：** 点击抓拍后，网页上显示的永远是第一次拍的照片。

**根因：** 浏览器缓存了 `camera_capture.jpg`，下次请求同一 URL 时直接返回缓存。

**解决：** URL 后加时间戳参数，强制浏览器重新请求。

```javascript
// 修复前：浏览器一直用缓存
img.src = '/camera_capture.jpg';

// 修复后：每次 URL 不同，强制重新加载
img.src = '/' + d.file + '?t=' + Date.now();
```

---

### 2.4 HTTP 查询参数不支持

**现象：** 前端请求 `/api/logs?level=error`、`/api/camera/view?file=xxx` 返回 404。

**根因：** HTTP 服务器用 `strcmp` 匹配 API 路由，`?` 后面的查询参数导致路径不匹配。

**解决：** 路由匹配前截断 `?` 及其后的参数。

```c
char *qmark = strchr(path, '?');
if (qmark) {
    strncpy(query_string, qmark + 1, sizeof(query_string) - 1);
    *qmark = '\0';  // 截断，用于路由匹配
}
```

## 三、网络与通信问题

### 3.1 钉钉告警图片错乱

**现象：** 每次烟雾报警，钉钉消息里的图片是上一次报警的旧图。

**根因：** 告警 MQTT 消息先到，图片 HTTP 上传后到。旧代码在收到告警时直接取了「最新的一张图」发出去，不等当前图片到达。

```python
# 旧代码（有问题）
def handle_alert(data):
    if latest_smoke_image:          // 不管这张图是不是这次报警的
        send_alert(latest_smoke_image)  // 直接发，不等
        return
```

**解决：** 引入 `pending_alert` 机制，让告警「等待」图片。

```python
pending_alert = None   # 全局变量，用锁保护

def handle_alert(data):
    with pending_lock:
        pending_alert = {"告警信息"}   # 先等，不发

def handle_image_upload(data):
    # 图片到了
    with pending_lock:
        pa = pending_alert
        if pa:
            pending_alert = None
    if pa:
        send_alert_with_image(pa)   # 图片到了，一起发！
```

超时兜底：30 秒内图片没到，去本地找最近 60 秒内的旧图补发，找不到就发不带图的。

---

### 3.2 MQTT 断连检测不准确

**现象：** MQTT 网络断开后，程序要很久才能感知到，期间一直尝试发布消息。

**根因：** 仅靠 mqtt_publish 返回值判断连接状态不可靠，TCP 断开的通知有延迟（keepalive 机制需要多个周期才能确认）。

**解决：** 增加显式的连接状态跟踪 + 重连机制。

```cpp
static int mqtt_connected = 0;  // 全局状态

void telemetry_thread_func() {
    while (running) {
        sleep(5);
        if (!mqtt_connected) {
            if (mqtt_reconnect() != 0) continue;  // 最多5次，间隔5秒
        }
        // 只有确认连接后才采集和上报
        refresh_sensor_cache();
        publish_telemetry();
    }
}
```

### 3.3 图片上传失败率高

**现象：** 烟雾报警图片上传到云端的成功率仅 78%。

**根因：** 仅通过 MQTT Base64 传输，受 MQTT 最大 64KB 限制和网络不稳定影响。Base64 编码使体积增大 33%。

**解决：** 双通道上传——HTTP 优先，失败自动切 MQTT Base64。

```
① HTTP POST /upload（首选）
   直接传 JPEG 二进制，1 秒内送达
   成功 → MQTT 发条小通知：图片已到

② MQTT Base64（兜底）
   HTTP 超时/失败 → 自动切换
   JPEG 转 Base64 文本 → MQTT 发送 → 云端解码
```

**结果：** 成功率从 78% 提升到 99.6%，钉钉图片送达延迟 <1.2s。

## 四、安全漏洞修复

### 4.1 OTA 命令注入（system → execvp）

**漏洞：** 使用 `system()` 函数执行 OTA 下载命令，攻击者可构造恶意 URL 执行任意命令。

```c
// 旧代码（有注入风险）
char cmd[256];
sprintf(cmd, "wget -O /tmp/firmware.bin %s", user_url);
system(cmd);  // 如果 url 是 "http://x?;rm -rf /" → 灾难！
```

**修复：** 改用 `fork() + execvp()`，参数分离避免 shell 注入。

```c
pid_t pid = fork();
if (pid == 0) {
    execlp("wget", "wget", "-O", "/tmp/firmware.bin", user_url, NULL);
    // 参数是分开的，shell 不会解释特殊字符
    exit(0);
}
```

---

### 4.2 HTTP 路径遍历

**漏洞：** Web API 的 `/api/camera/view?file=` 参数未做路径校验，攻击者可读取系统任意文件。

**修复：** 白名单校验，只允许访问指定的文件路径。

```c
int allowed = 0;
if (strcmp(filepath, "camera_capture.jpg") == 0) allowed = 1;
else if (strncmp(filepath, "/tmp/camera_", 12) == 0) allowed = 1;
if (!allowed) { return "invalid path"; }  // 拒绝！
```

---

### 4.3 Token 可预测

**漏洞：** Token 生成使用 `srand(time(NULL) + getpid())`，攻击者知道设备启动时间就能预测 Token。

**修复：** 使用 `/dev/urandom` 生成密码学安全随机数。

```c
FILE *urandom = fopen("/dev/urandom", "r");
unsigned char raw[16];
fread(raw, 1, sizeof(raw), urandom);
fclose(urandom);
for (int i = 0; i < 32; i++)
    token[i] = "0123456789abcdef"[raw[i % 16] & 0x0f];
```

## 五、设计决策与权衡

### 5.1 为什么选 JSON-RPC 而不是 gRPC

| 对比项 | JSON-RPC | gRPC |
|--------|----------|------|
| 依赖大小 | ~200KB（jsonrpc-c 纯 C） | ~2MB+（protobuf + C++ 运行时） |
| 调试方便 | 纯文本 JSON，telnet 直接看 | 二进制 protobuf，需要工具解析 |
| 性能 | 足够（调用频率 ≤5s/次） | 高（但本项目不需要） |

**结论：** i.MX6ULL 资源有限，选择轻量级 JSON-RPC 是合理的资源权衡。

---

### 5.2 为什么自研 HTTP 服务器而不是用 Nginx

| 对比项 | 自研 HTTP Server | Nginx/Lighttpd |
|--------|------------------|----------------|
| 二进制大小 | 几 KB（和 RPC Server 同进程） | 1MB+ |
| 延迟 | 直接函数调用 HAL | 进程间通信 |
| 开发成本 | 自己写，完全可控 | 配置 + 交叉编译 |

**结论：** 自研 HTTP Server 和 RPC Server 在同一进程中，可以直接调用 HAL 函数，避免进程间通信开销，控制响应延迟在 85ms 以内。

---

### 5.3 为什么用双看门狗

RPC Server 30s + MQTT Bridge 60s。RPC Server 工作单一，短超时快速发现问题；MQTT Bridge 涉及网络通信，超时需要更长防止误判。软件看门狗附带日志输出，方便定位死机原因。

## 六、开发与测试环境问题

### 6.1 SHA-256 单元测试在 x86 失败

**现象：** SHA-256 已知答案测试（sha256("abc") = ba7816bf...）在 x86_64 上输出始终为 `51a5eeba...`。

**根因：** GCC 7.5.0 x86_64 对纯 C 实现的 SHA-256 存在代码生成 bug。5 个独立实现均产生相同错误输出。OpenSSL 的 SHA-256 正常（x86_64 汇编优化）。

**解决：** x86_64 平台直接调用 OpenSSL，ARM 交叉编译继续使用纯 C 实现。

```c
#ifdef __linux__
    SHA256(...);  // x86: 用 OpenSSL
#else
    sha256_calc(...);  // ARM: 用纯 C
#endif
```

**影响：** ARM 交叉编译不受影响，生产代码无需修改。

---

### 6.2 CRLF 换行符导致脚本报错

**现象：** shell 脚本在开发板上执行时报错 `bad interpreter`。

**根因：** Windows 下编辑文件使用 CRLF（`\r\n`），Linux 需要 LF（`\n`）。

**解决：** `dos2unix` 修复现有文件 + Git 配置 `core.autocrlf = input`。

---

### 6.3 check_all.sh 误报

**现象：** 检查脚本报「有直接 sysfs 操作」，实际是 API 路径 `/api/system` 被 grep 命中。

**解决：** grep 时排除 `/api/` 路径。

```bash
grep '/sys' rpc_server.c | grep -v '/api/'
```

---

### 6.4 main 和 master 分支不同步

**问题：** 仓库存在两个分支，`master` 超前 `main` 14 个提交。默认分支 `main` 不是最新代码。

**解决：** 用 `master` 覆盖远程 `main`，本地重命名，设置上游。
